# Security Baseline

This document defines the minimum security requirements for the online hotel
booking application. It applies to all application code, APIs, background jobs,
operational tooling, and third-party integrations.

<!-- markdownlint-disable MD013 -->

## Scope and principles

- Security controls must be enforced server-side; client-side checks are usability aids only.
- Deny by default when identity, authorization, validation, or payment safety requirements are ambiguous.
- Store and log only the data required to operate, support, and audit the booking workflow.
- Prefer battle-tested platform libraries and managed providers over custom cryptography, session, or payment handling.

## Authentication

### Password hashing

- Passwords must never be stored, logged, emailed, or returned in plaintext.
- Hash passwords with a modern, adaptive password hashing algorithm such as Argon2id or bcrypt.
- Use per-password salts generated by the hashing library; do not reuse or hard-code salts.
- Configure work factors to meet current platform guidance and review them at least annually.
- Rehash passwords on successful login when the stored hash uses an outdated algorithm or work factor.
- Password reset tokens must be single-use, high entropy, short lived, and stored only as hashes.

### Login and account recovery

- Return generic authentication errors so attackers cannot enumerate accounts.
- Apply application-level rate limits to login, registration, password reset, and MFA challenge endpoints.
- Require email verification before enabling sensitive account actions.
- Use step-up authentication before changing passwords, email addresses, payout settings, payment methods, or administrator roles.

## Session and token rules

- Session identifiers, access tokens, refresh tokens, and reset tokens must be generated with a cryptographically secure random source.
- Browser session cookies must use `HttpOnly`, `Secure`, and `SameSite=Lax` or stricter unless a documented integration requires otherwise.
- Access tokens must be short lived. Refresh tokens and long-lived sessions must be revocable and rotated after use.
- Session expiration must include both idle timeout and absolute lifetime limits.
- Invalidate all active sessions after password change, account compromise, credential reset, or role downgrade.
- Bind session validation to server-side state so disabled users, revoked roles, and terminated sessions take effect immediately.
- Never put session tokens, reset tokens, or API keys in URLs.

## Authorization and RBAC

Authorization must be enforced on every server-side route, API resolver, job entry point, and administrative action. UI hiding is not authorization.

### RBAC matrix

| Role | Typical users | Allowed capabilities | Explicit exclusions |
| --- | --- | --- | --- |
| Guest | Unauthenticated visitor | Search hotels, view public room information, start registration or login | Create bookings, view private booking data, access account or admin APIs |
| Customer | Registered booking user | Manage own profile, create and cancel own bookings within policy, view own invoices and payment status | View or modify other users' bookings, change room inventory, access admin reports |
| Hotel Staff | Property operations user | View assigned property bookings, update check-in/check-out status, manage assigned room availability | Access unrelated properties, change user roles, view full payment data |
| Support Agent | Customer support user | Search customer records for support, assist with booking changes according to policy, view audit-relevant account status | Export bulk sensitive data, access full card data, grant admin roles |
| Administrator | Trusted platform operator | Manage users, roles, properties, inventory, policy settings, and operational reports | Bypass audit logging, access plaintext secrets, handle raw card data |
| Service Account | Internal system integration | Perform the narrowly scoped machine action assigned to the credential | Interactive login, broad administrative access, use outside documented integration scope |

### Authorization requirements

- Use least privilege for every role and service account.
- Check object ownership and tenant/property boundaries on every read, update, and delete operation.
- Centralize authorization logic where practical to avoid inconsistent route-level decisions.
- Record role changes, permission grants, and permission revocations in the audit log.
- Review privileged accounts and service accounts on a recurring schedule.

## CSRF and XSS considerations

### CSRF

- State-changing browser requests must be protected with CSRF tokens, same-site cookie settings, or another framework-approved anti-CSRF control.
- CSRF tokens must be unique per session, unpredictable, validated server-side, and excluded from logs.
- Do not allow unsafe state-changing operations via `GET` requests.

### XSS

- Escape output by context using framework defaults for HTML, attributes, JavaScript, URLs, and CSS.
- Sanitize any user-generated rich text with an allowlist sanitizer before storage or rendering.
- Use a restrictive Content Security Policy that avoids unsafe inline scripts where practical.
- Store authentication tokens in `HttpOnly` cookies rather than browser-accessible local storage.
- Validate uploaded files by type and size, store them outside executable paths, and serve them with safe content headers.

## Server-side validation

- Validate every request on the server, including requests from first-party clients and internal tools.
- Use allowlist schemas for request bodies, query parameters, path parameters, headers, and webhooks.
- Enforce type, length, range, enum, format, ownership, and business-rule constraints.
- Reject unknown or unexpected fields for sensitive operations.
- Normalize input before validation when normalization affects security decisions.
- Use parameterized queries or safe ORM APIs for all database access.
- Validate third-party webhook signatures and timestamps before processing events.

## Application-level rate limiting

- Rate limit authentication, password reset, registration, booking creation, search, promo-code validation, payment initiation, and support/admin endpoints.
- Apply limits by a combination of user ID, IP address, session, device fingerprint where available, and target account for account recovery flows.
- Prefer progressive throttling and temporary lockouts over permanent denial.
- Return generic rate-limit responses that do not reveal whether an account exists.
- Log rate-limit events with enough context for abuse investigation while redacting secrets and payment data.
- Document operational overrides and require privileged approval for allowlist changes.

## Audit logging

Audit logs must be tamper-resistant, access controlled, searchable for incident response, and retained according to the data retention policy.

Log at minimum:

- Login success and failure, logout, MFA challenge outcomes, and session revocation.
- Password reset requests and completions without logging reset tokens.
- Role, permission, and service account changes.
- Booking creation, cancellation, refund initiation, and policy override decisions.
- Administrative access to customer records and exports.
- Payment initiation, payment status transitions, refund status transitions, and payment provider webhook processing.
- Security control events such as validation failures, CSRF failures, rate-limit enforcement, and suspicious access denials.

Each audit event should include timestamp, actor, action, target resource, outcome, request correlation ID, and source context such as IP address or service identity when available.

## Sensitive data handling

- Classify personal data, credentials, tokens, secrets, payment metadata, and operational logs before storage or processing.
- Encrypt sensitive data in transit using TLS and at rest using managed encryption controls.
- Keep secrets in a managed secrets store; never commit secrets to source control or place them in client bundles.
- Limit access to production data by role and record privileged access in audit logs.
- Define retention and deletion schedules for personal data, audit records, and support exports.
- Use masked values in administrative interfaces unless full value access is explicitly required and audited.

### Sensitive log redaction

Logs, traces, metrics, analytics events, and error reports must redact or omit:

- Passwords and password hashes.
- Session IDs, access tokens, refresh tokens, reset tokens, API keys, and CSRF tokens.
- Full payment card numbers, CVV/CVC, magnetic stripe data, PINs, and bank account numbers.
- Government identifiers and other regulated personal data unless explicitly approved.
- Raw request or response bodies for authentication, payment, and account recovery endpoints.

Use structured logging with field-level redaction. Redaction must happen before data leaves the application process for log aggregation or third-party monitoring.

## Payment safety

- Use a PCI-compliant payment provider for card entry, tokenization, authorization, capture, and refunds.
- Do not store, process, transmit, or log raw cardholder data unless a formally approved PCI scope requires it.
- Exclude the following from application databases and logs: full PAN, CVV/CVC, magnetic stripe data, card PIN, and equivalent raw payment credentials.
- Store only provider tokens, payment intent identifiers, masked card brand/last-four/expiry when needed, status, amount, currency, and reconciliation metadata.
- Verify payment webhooks with provider signatures and idempotency keys before updating booking or refund state.
- Treat payment status from the provider as authoritative; never mark a booking as paid from client-side confirmation alone.
- Reconcile booking, invoice, refund, and provider states through idempotent server-side workflows.

## Review cadence

- Review this baseline when authentication, authorization, payment, logging, or data handling architecture changes.
- Review at least annually for changes in framework defaults, regulatory expectations, password hashing guidance, and payment provider requirements.
- Security exceptions must be documented with owner, risk, mitigation, expiration date, and approval.
